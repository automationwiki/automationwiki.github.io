B.        GitLab-Github Hooks

  1. When any code is pushed to the Master branch, the development Jenkins build starts after that dev build is finished, then our Suite runs should start accordingly.
  2. In Local Jenkins we have a Job configured to run the Suites
  3. We need a public IP in order to implement gitlab webhooks in jenkins .
  4. Once a single Job is triggered, then based on the same trigger we can start other jobs in the same machine as well as other Node machines.

C.        Gitlab - Web Hooks

  1. URL: public url of the Jenkins job.
  2. Secret Token: Any token to identify.
  3. Trigger: Push events, Tag push events, Comments, Confidentials Comments, Issues events, Confidential Issues events, Merge request events, Job events, Pipeline events, Wiki Page events.
  4. SSL Verification: Optional.

  2. Then, for Master Machine Job, we can add the gitlab webhook, so that when the webhook is triggered, the master job knows what all jobs to start the execution.
  3. Build when a change is pushed to GitLab. GitLab webhook URL: http://localhost:8080/project/WEB_REG_Guest_Beta or http://169.254.98.206:8080/
  4. Enabled GitLab triggers
    - Push Events
    - Push Events in case of branch delete
    - Opened Merge Request Events
    - Build only if new commits were pushed to Merge Request
    - Accepted Merge Request Events
    - Closed Merge Request Events
    - Rebuild open Merge Requests: Never, On Push to Source Branch, On push to source or target branch.
    - Approved Merge Requests (EE-only)
    - Comments
    - Comment (regex) for triggering a build: Jenkins please retry a build

J.1        Jenkins Instalation

  1. Reference: https://wiki.jenkins.io/display/JENKINS/Installing+Jenkins+as+a+Windows+service
  2. Download from: https://www.jenkins.io/download/
  3. Download the LTS (Stable Version):
    Generic Java package: (jenkins.war)-if you wish to run jenkins as a java command: Navigate to the war location and run  “java -jar jenkins.war” in command prompt. Once Jenkins is started this way, look for the "Install as Windows Service" link in the "Manage Jenkins" page at http://<hostname>:8080/ and Provide Already existing Path where you want to run jenkins service. E.g. C:\Jenkins (requires Microsoft .NET framework version 2.0): Note: Choose the directory where Jenkins shall be installed (directory must already exist. If not create it beforehand). This will become JENKINS_HOME and used to store data files and programs alike.
  4. Upon successful completion of the installation, you should see a page asking you to restart Jenkins. This is to re-launch Jenkins as a newly installed Windows service. When you click yes, you'll be asked to wait until the restart completes:
  5. Note: If a restart fails for some reason, check the output from Jenkins, which is stored in the installation directory that you specified. For W2K8 R2 users you may need to add C:\Windows\SysWOW64 to the PATH env var.
  6. At this point you can use the service manager to confirm that Jenkins is running as a service. (services.msc)
  7. Stable Installer for Windows: (jenkins.msi)- if you want to Install Jenkins as a service.
  8. NOTE: if you installed Jenkins using the windows installer, you shouldn't need to do anything else here because the windows installer automatically runs Jenkins as a windows service.
  9. Now connect to Jenkins by going to the following URL:  http://<hostname>:8080/
  10. If you are running in Localhost then url will be: http://localhost:8080/
  11. NOTE: The "Install as Windows Service" link indicated above will not appear if Jenkins is already running as a service on the machine. You can verify that Jenkins is running as a service by launching the Services app from the Windows Start Menu and looking for a running service called "jenkins" among the list of all windows services running on the machine.

J.2        Install Slave as a Windows service (require .NET 2.0 framework)

  1. Reference: https://scmquest.com/jenkins-master-slave-setup-and-configuration-with-screenshots/
  2. Since each agent runs a separate program called an "agent" there is no need to install the full Jenkins (package or compiled binaries) on an agent. There are various ways to start agents, but in the end the agent and Jenkins master need to establish a bi-directional communication link (for example a TCP/IP socket) in order to operate.
  3.  Java Network Launch Protocol.
  4. Go to Jenkins dashboard -> Manage Jenkins -> Manage Nodes.
  5. Name the node and click on OK.
  6. Note: Initially you will get only one option “Permanent Agent” or “Dumb Slave“(in earlier jenkins versions), but once you have one slave then you will get another option to copy the node from another node which named as “Copy Existing Node”
  7. Name : Name of the Slave which should be unique.
  9. Description : Optional , Description for this slave, but as I said earlier, it would be really helpful for other team members
  10. # of executors : The maximum number of concurrent builds that Jenkins may perform on this agent. I have used just 1 executor for testing purpose, but a good value to start with would be the number of CPU cores on the machine. You can check the server stat and then define the number of executors.
  11. Remote root directory : An agent needs to have a directory dedicated to Jenkins. Specify the path to this directory on the agent. It is best to use an absolute path, such as /var/jenkins/node1 or c:\jenkins\node1. This should be a path local to the agent machine. There is no need for this path to be visible from the master.
  12. Labels : Labels (or tags) are used to group multiple agents into one logical group. Multiple labels must be separated by a space. For example, linux docker would assign two labels to the agent: linux and docker.
  13. Usage : Controls how Jenkins schedules builds on this node. Utilize this node as much as possible- This is the default and normal setting. In this mode, Jenkins uses this node freely
  14. Launch method : It Controls how Jenkins starts this agent.
  15. – Launch agent via Java Web Start: It allows slave to be launched using Java Web Start. In this case, a JNLP file must be opened on the agent machine, which will establish a TCP connection to the Jenkins master. This means that the agent need not be reachable from the master; the agent just needs to be able to reach the master. If you have enabled security via the Configure Global Security page, you can customize the port on which the Jenkins master will listen for incoming JNLP agent connections. By default, the JNLP agent will launch a GUI, but its also possible to run a JNLP agent without a GUI, e.g. as a Window service.
  16. – Launch agent via execution of command on the master: It starts slave by having Jenkins execute a command from the master. Use this when the master is capable of remotely executing a process on another machine, e.g. via SSH or RSH.
  17. – Launch slave agents via SSH: It starts a slave by sending commands over a secure SSH connection. The slave needs to be reachable from the master, and you will have to supply an account that can log in on the target machine. No root privileges are required. This is the one which i am using for my slave configuration.
  18. — Host Key Verification Strategy : Controls how Jenkins verifies the SSH key presented by the remote host whilst connecting.
  19. — Known Host file verification strategy : Checks the known_hosts file (~/.ssh/known_hosts) for the user Jenkins is executing under, to see if an entry exists that matches the current connection.
  20. — Mannually provided key verification strategy : Checks the key provided by the remote host matches the key set by the user who configured this connection.
  22. — Mannually trusted key verification strategy : Require a user with Computer.CONFIGURE permission to authorise the key presented during the first connection to this host before the connection will be allowed to be established.
  24. — Non Verifying verification strategy : Does not perform any verification of the SSH key presented by the remote host, allowing all connections regardless of the key they present. Its not advisable to select as it may open the path for attackers.
  26. – Let Jenkins control this Windows slave as a Windows service: It starts a Windows slave by a remote management facility built into Windows. Suitable for managing Windows slaves. Slaves need to be IP reachable from the master.
  28. Availability : Controls when Jenkins starts and stops this agent.
  30. Keep this slave on-line as much as possible : This one is the default and normal setting. In this mode, Jenkins tries to keep the slave on-line as much as possible. If Jenkins can start the slave without user assistance, it will periodically attempt to restart the slave if it is unavailable. Jenkins will not take the slave off-line.
  32. Take this slave on-line when in demand and off-line when idle : In this mode, if Jenkins can launch the slave without user assistance, it will periodically attempt to launch the slave while there are unexecuted jobs else the slave will be taken off-line by Jenkins
  33. You can also define the node properties, like environment variables, such as PATH, JAVA_HOME, etc and Tools locations such as, ANT, GIT, etc.
  34. Environment variables defined here will be made available to every build executed by this agent, and will override any environment variables that have the same Name as those defined on the Configure System page.
  36. For Tool Locations, you can specify the location of certain tools on this node, overriding the global configuration (You may prefer to use automatic tool installers instead, removing the need to configure each node separately)
  38. Once you are done click on SAVE and that’s it. Your new Jenkins Slave is up and running and ready to take the builds
