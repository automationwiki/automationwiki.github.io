SLNO CATAGORY GUIDELINES
1 Overview Coding Guidelines to be followed for development of software and following sections describe the approach to be followed for Java Projects in general and specific to Test Automation.
It helps readability of the code to understand the functionality and logic, it helps get consistency across projects in reviewing the code and most importantly it helps to maintain and do rework easily.
This document shall be used as reference for Voot Test Automation while programming and as checklist for code review process.
2 Naming conventions

Package Name Package names should be in all lower cases that start with reversed Internet domain name followed by application name. For example: org/companyname/appname

Class names Upper camel case should be used for the class names. Try to make the name simple and descriptive.


LoginTest.java : All the tests shall have suffix Test


LoginPage.java : All the pages shall have suffix Page


BaseTest.java : Class to include all common methods to be used by tests and pages


BasePage.java : Class to include all common methods to be used by tests and pages

Method names Lower camel case should be used for the method names. Try to make the name simple and descriptive. Use whole word instead of abbreviations.


initialiseBrowsers()


initialiseProperties()


waitForTextToBePresent(WebElement webElement, String textToBeVerified)

Variable names Lower camel case should be used for the variable names. Intent of the variable shall be indicated with the given name. The variable name should be a noun. Try to make the name simple and descriptive. Try to avoid abbreviations, prefer domainCode over dmnCd


webDriver


intEmployeeCount

Constants All letters should be in UPPER_CASE. Words shall be separated with underscore _ . Create a constant file for static values that are needed at multiple places


Use Database-driven values for dynamic values


Use ENUMs for a group of constants


BASE_URL


All the global variables and Constants shall be defined in respective class files and loaded into the constants defined in Constant class.


Constants shall be defined as public static in the Constant class


public static String BASE_URL;
3 Programming guidelines

Optimize Imports Remove unused imports before pushing the code.

Null Checks As a general practice, always do a null check on a variable before any operation. ∙ Exception Handling


The try-catch block should be used for exception handling with proper logging in the catch block. Also, make sure to close the resources properly in the finally block.

Code Indentation and Formatting - Code Review Always format your class before creating a PR for an easy Review Process.


It can be done automatically with the built-in editor of the IDE. For instance, use Ctrl-Shift-F in Eclipse.

Not All One-Liners It's good to keep the code clean and readable. So, it's a better idea to not always go with one-liners. Especially, when we initialize and operate the variable in one line.


For example, write:


out.write(attrs.get("offset") + "-" + Math.min(attrs.get("count"), attrs.get("offset") + attrs.get("max")) + " " + title + " " + attrs.get("count"));


as:


int start = attrs.get("offset")


int total = attrs.get("count"))


int end = Math.min(total, start + attrs.get("max"))


out.write(start + "-" + end + " " + title + " " + total)

∙ White-Spaces Use white-spaces to separate combined statements to make code more readable. For example, write:


Integer.valueOf(params.offset?params.offset:attrs.offset)


as:


Integer.valueOf(params.offset ? params.offset : attrs.offset)

∙ Curly Braces in If Loops: Use curly braces for one-liners also. Braces to be used with if, else, for, do and while statements, even when the body is empty or contains only a single statement.

∙ One statement per line Each statement is followed by a line break.

∙ Variable declarations In For Loop, loop variables should not be a single letter like i, or j. It should be a noun.

∙ StringBuilder or StringBuffer in Place of String When performing a lot of operations on the String, use StringBuilder or StringBuffer


For example, Java creates a new String object for every concatenation operation. In this case, a better idea is to use a StringBuffer.


Use equals over ==


equals perform the actual comparison of two strings, whereas == compares object references. ∙

switch-case Over Multiple if-else It's a good practice to use switch-case in place of multiple if-else conditions. It optimizes the code execution and also makes code cleaner and more readable.

∙ Objects Creation in a Loop It is usually better to create the object inside the loop (If object is not required outside loop). Java optimizes memory usage for short-lived objects.

For example, write: Person person;
for(int i=0;i<namesList.size();i++) {
person = new Person();
person.setName(namesList.get(i));
person.display();
}

as: for(int i=0;i<namesList.size();i++) {
Person person = new Person();
person.setName(namesList.get(i));
person.display();
}


Also, create a new object only if required.

For example, write: ArrayList<Person> personList = new ArrayList<Person>();
for(int i=0;i<namesList.size();i++) {
Person person = new Person();
if(null != namesList.get(i)) {
person.setName(namesList.get(i));
personList.add(person);
}
}

as: ArrayList<Person> personList = new ArrayList<Person>();
for(int i=0;i<namesList.size();i++) {
if(null != namesList.get(i)) {
Person person = new Person();
person.setName(namesList.get(i));
personList.add(person);
}
}
4 Test Automation Best Practices


Follow Page Object Design Pattern Page Objects should never make verification or assertions, but there is a single verification which can and should be within the page object.

Waits Usage of Implicit Waits, Explicit Waits and Sleeps should be optimal in the project. For example a 10 seconds wait at 12 places in a class, across 60 different classes can add 2 hours to the execution time.

Locators Strategy While Xpath is convenient to find any element in the DOM, it is considered as least preferred approach with respect to performance. In iOS platform, Xpath should be avoided to the extent possible.
5 Comments Author Name is Mandatory to Mention in Each class like


// Author:


//Last Modified by:


Each Method should have a description in comments. Mandatory.


Comments shall be used to explain the code to make the test scripts more readable for others. Always put comments defining the purpose.


Use consistent commenting styles.

Block comment style Block comments are indented at the same level as the surrounding code. They may be in /* ... */ style or // ... style. For multi-line /* ... */ comments, subsequent lines must start with * aligned with the * on the previous line.


/*
* This is // And so /* Or you can
* okay. // is this. * even do this. */
*/


Comments are not enclosed in boxes drawn with asterisks or other characters.


When writing multi-line comments, use the /* ... */ style if you want automatic code formatters to re wrap the lines when necessary (paragraph-style). Most formatters don't re-wrap lines in // ... style comment blocks.

For example, Javadoc on a class: /**
* General convenience tags for layout - header, body and footer
* @author – Name
* @dateCreated - Date
*/
class LayoutTagLib {

}

Javadoc on a method: /**
* Gets the user for specified code and role.
* @param code :- The code, either username or email address
* @param role :- The role identification e.g. A, B or C. Default is A.
* @return the user or null if not found
*/
User findUser(String code, String role = "A")


Make sure the code is self-explanatory and comments are really useful in very specific cases.
6 TestNG Annotations
Structure the Test methods for flexible suite run options by using annotations with required tags like @Test, @enabled=true, @groups, @priority etc as decided by the project.


Ensure @Before and @After methods are consistently followed at Suite Level, Class Level and Method Level
7 Code Clean Up
Remove console print Statements (SOPs), use logging instead (never log personal information) Remove obsolete comments


Use the @deprecated annotation on the method/variable, if it is not meant for future use or going to be removed


Remove hardcoded variable values
8 Business Logic and reusability
Avoid redundant code by using reusable components like utilities and Event Utils methods.
9 Code Commit
Group the files and commit together for a logical set of changes. Never miss any change that is dependent for other classes, else others will get errors in their project.


Commit messages should contain the task information. For example, JIRA issue number, a meaningful comment on code implementation
10 Code Review and Comments.
Source code maintained at GitHub as a private repository. To ensure that, comments are added for the code change that is being pushed. Comments are required for Reviewers to understand the purpose of change.


Discuss with leads for any change in common Utilities Classes. Requires approval from leads to push any changes in Utilities.


Before start of the day, to take the latest code from GitHub. Need to Merge the Code as soon as possible after getting approvals else it might result in conflicts.


Always save all files before pushing the code. Don’t push the Constant file and those changes that are local to developers system.


Don’t push the code, even if there is a single error in the project.


In case of conflicts, resolve and create a pull request again.


Technical Lead or Module Leads approving pull requests, to ensure coding guidelines are followed. Reviewers to record any observation and gaps to the coding guidelines and to get it reworked from developers, before approving pull request and merging the change set.


Keep It Simple! Maintain simplicity and readability of code.
